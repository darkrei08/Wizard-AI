# 🤖 Workflow: Hierarchical Agent Management

This meta-workflow outlines how the primary (orchestrator) AI agent should manage, delegate, and review work performed by parallel or sequential subagents. This applies to complex tasks that are broken down during the planning phase.

## 🧬 The Agent Hierarchy

1. **Orchestrator Agent (You)**: The primary agent that communicates with the user, creates the `implementation_plan.md`, and tracks progress in `task.md`.
2. **Specialized Subagents**: Independent agent instances spawned by the Orchestrator (via `subagent-driven-development` or native platform tools like `invoke_subagent`).

## 🔄 The Delegation Loop

When executing a complex plan, you MUST follow this loop:

### 1. Task Definition (`dispatching-parallel-agents`)
- For each task in `task.md`, prepare a strict, highly specific prompt.
- The prompt must specify exactly what files to edit, what tests to write, and what the success criteria are.
- Mention specific skills the subagent must load (e.g., "Use the `react` and `taste-skill` frameworks").

### 2. Spawning Subagents
- Launch the subagent(s). If tasks are independent (e.g., building 3 separate React components), spawn them in parallel.
- If tasks are sequential (e.g., build the database schema, then the API), wait for the first to complete.
- Do NOT let subagents communicate with the user directly.

### 3. Verification & Review (`receiving-code-review`)
- Once a subagent finishes, the Orchestrator MUST review the work.
- Run `wz-ai-debug` or equivalent test commands to ensure the subagent didn't break the build.
- If the work is flawed, spawn a new debugging subagent or fix it yourself (`systematic-debugging`).

### 4. Integration
- After successful review, mark the task as `[x]` in `task.md`.
- Ensure all context generated by the subagent is summarized into `MEMORY.md` to prevent context bloat.

## ⚠️ Core Rules
- **Never do all the work yourself** if the task is complex. Delegate.
- **Never trust subagent output blindly.** Always run verification commands before reporting success to the user.
